Mental model

Front- en back end
Voor ik aan deze minor begon had ik in principe helemaal geen kennis over onderwerpen zoals: programmeren, coderen, HTML/CSS, front- en back end. Ik heb ze weleens gehoord, maar ik was nooit in staat deze duidelijk te kunnen uitleggen aan mezelf, laat staan iemand anders. Ik heb dus pas kennis gemaakt met deze begrippen toen ik aan deze minor begon.

Als programmeur kun je je specialiseren in front end en/of back end programmeren. Op het moment dat je beide onder de knie hebt mag je je zelf full-stack developer noemen. Wat is nu het verschil tussen front end en back end en wat betekenen ze in de eerste instantie?

Stel je hebt een ‘simpele’ webshop. Op het moment dat je naar deze website gaat in je browser wordt alle code van de front end gedownload en wordt deze weergegeven op je beeldscherm. Alles wat je op deze pagina ziet is dan in principe front end.
Back end komt om de hoek kijken op het moment dat er bijvoorbeeld een account moet worden aangemaakt of er een bestelling wordt geplaatst. Je zou kunnen zeggen; zodra er informatie voor een langere tijd moet worden opgeslagen, je niet meer genoeg hebt aan alleen front end. De front- en back end praten constant met elkaar zodra de gebruiker van een website een actie uitvoert. Op het moment dat de gebruiker bijvoorbeeld een account aanmaakt dan stuurt de front end een bericht naar de back end waarin bijvoorbeeld staat: ‘nieuwe user aanmaken, gebruikersnaam=voorbeeld123’. De back end krijgt deze melding binnen, gaat hiermee aan de slag en stuurt vervolgens een bericht terug naar de front end met bijvoorbeeld= ‘Gebruiker is aangemaakt’ of ‘Gebruiker is reeds al bekend’.

Als front end programmeur houdt je je vooral bezig met de interactie van een gebruiker en het visuele aspect van de applicatie of website. Als back end developer ben je meer bezig met het verwerken van data en het opslaan van deze data in een database.

Database
No-SQL is de verzamelnaam voor ‘tegenhangers’ van de SQL-standaard op het gebied van databases. De term No-SQL wordt vaak verkeerd begrepen. No-SQL betekent namelijk ‘Not Only SQL’. Het is dan ook geen vervangend product voor SQL, maar meer een aanvulling op relationele databases. Er zijn een aantal verschillen tussen SQL- en No-SQL databases. No-SQL databases verschillen van traditionele SQL/relationele databases in het opzicht dat ze geen relationeel model gebruiken. Ze werken vaak schema loos en stellen daarom weinig eisen aan de organisatie en structuur van de data. No-SQL is eenvoudig via een API aan te sturen en is in tegenstelling tot SQL horizontaal schaalbaar. Met horizontaal schaalbaar wordt er bedoeld dat de data gesplitst kan worden tussen diverse servers. Bij SQL databases kan de data alleen verticaal geschaald worden wat betekent dat er meer resources in een server geplaatst moet worden om de snelheid te vergroten. No-SQL is een stuk eenvoudiger in gebruik en kan kostenbesparend werken. Een nadeel is de onnauwkeurigheid van No-SQL.

In een SQL database kunnen gegevens op een gestandaardiseerde manier worden opgeslagen, buiten de applicatie die de gegevens gebruikt. Een groot nadeel van deze databases is dat zij niet uitblinken in het verwerken van grote hoeveelheden gegevens of het serveren van gegevens onder hoge druk, tenzij er fors geïnvesteerd wordt in hardware.

API's
Een API is een afkorting die staat voor ‘Application Programming Interface’. Een API maakt het mogelijk om andere systemen en software een ingang tot een ander systeem te bieden. Hiermee kunnen verschillende systemen met elkaar communiceren en gegevens uitwisselen.

Een API is volledig flexibel te ontwikkelen. Dit betekent dat er in principe geen limieten zitten aan hoe je deze API wilt ontwikkelen. Op het moment dat twee systemen via het internet met elkaar moeten ‘spreken’ wordt er gebruik gemaakt van een webservice. Webservices zijn API’s op afstand. Een API kan dan worden aangevraagd via een specifieke URL die dan een XML of JSON formaat terug geeft.
